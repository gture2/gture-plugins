# Documentation Update Report Template

This template defines the structure for the summary comment posted at the end of a `doc-writer` run. The orchestrator must follow this format exactly.

---

## Variant A — Documentation Updates Were Applied

```markdown
## 📝 Documentation Update Summary

> A companion documentation PR has been opened: **[#<DOCS_PR_NUMBER>](<DOCS_PR_URL>)** — targets `[head branch]`.
> Merge it into `[head branch]` before merging this PR so the docs land alongside the code.

**PR:** #[number] — [title]
**Feature branch (docs PR target):** `[head branch]`
**Docs branch (docs PR head):** `docs/pr-[number]-sync`
**Docs PR:** [#<DOCS_PR_NUMBER>](<DOCS_PR_URL>)
**Doc-update commit:** [`<sha>`]
**Files changed:** [N] documentation file(s)

| Disposition | Count |
|---|---|
| ✏️ Updated | [N] |
| ➕ Added | [N] |
| ➖ Removed | [N] |
| 🔁 Renamed references | [N] |
| ⏭️ No-op (internal change) | [N] |

---

### ✏️ Updated

> Existing documentation that drifted from the new code

| Doc file | Section / symbol | Reason |
|---|---|---|
| `docs/api/users.md` | `POST /users` payload | New required field `tenantId` |
| `README.md` | Configuration | Renamed env var `DB_URL` → `DATABASE_URL` |
| … | | |

*(If none: "No existing documentation needed an update.")*

> New documentation for newly added public surfaces

| Doc file | Entry | What was added |

| `docs/cli/commands.md` | `mycli sync` | New subcommand added in this PR |
| `CHANGELOG.md` | `[Unreleased] → Added` | Public API for X |
| … | | |

*(If none: "No new documentation entries were required.")*

---

### ➖ Removed

> Documentation for behaviour that no longer exists

| Doc file | Entry | Why removed |
|---|---|---|
| `docs/api/legacy-export.md` | (whole file) | Endpoint removed in this PR |
| … | | |

*(If none: "No documentation entries were removed.")*

---

### 🔁 Renamed References

> Renamed symbols, commands, env vars, routes, or files swept across the doc tree

| Old name → New name | Doc files updated |
|---|---|
| `getUser` → `fetchUser` | `docs/api/users.md`, `README.md` |
| … | |

*(If none: "No renames in this PR.")*

---

### ⏭️ No-Op — Source Changes That Required No Doc Update

> Files in the PR diff that were assessed and determined to need no documentation work

| Changed file | Why no doc change |
|---|---|
| `src/internal/cache.ts` | Internal helper, no public surface |
| `src/users.test.ts` | Test file |
| `package-lock.json` | Lockfile |
| … | |

*(Only list the most informative entries — if the list would exceed ~15 rows, collapse the rest into "and [N] more internal / test / build files.")*

---

### ⚠️ Documentation Gaps — Please Review

> Source changes that may require documentation, but where the right home for the new content is unclear, OR where the change has implications beyond what an automated update can decide

- _Example: `src/billing/proration.ts` introduces a new proration algorithm. The behavioural change is significant but there is no existing billing documentation file — please decide where this should live._
- _Example: PR removes the `--legacy-output` CLI flag. Existing docs describe it, but the repo's deprecation policy is unclear — please confirm whether to remove the docs outright or mark them as deprecated._

*(If none: "No documentation gaps detected.")*
```

## Variant B — No Documentation Updates Were Required

Use this shorter variant when every changed file in the PR was assessed as **No-op** (purely internal, refactor, test, formatting, build, or dependency change with no user-visible effect).

```markdown
## 📝 Documentation Update Summary

**PR:** #[number] — [title]
**Branch:** `[head branch]`
**Result:** No documentation updates required.

All [N] changed file(s) were internal / test / build changes with no impact on the project's public surface, configuration, CLI, API, routes, or behaviour described by the existing documentation.

<details>
<summary>Files reviewed</summary>

| File | Why no doc change |
|---|---|
| `src/internal/cache.ts` | Internal helper, no public surface |
| `src/users.test.ts` | Test file |
| … | |

</details>

---

*Generated by [doc-writer](https://github.com/xianix-team/xianix-plugins-official) · [timestamp]*
```

## Notes on populating the template

- **Docs PR link** — Variant A **must** include `DOCS_PR_NUMBER` and `DOCS_PR_URL` in the top callout, in the metadata block, and in the closing footer. The whole point of this comment is to direct the reviewer to the docs PR.
- **Commit SHA** — use the short SHA (first 7 chars) for the inline link, full SHA for any commit references.
- **File paths** — always render in inline code (\`like-this\`).
- **PR title** — use the title as fetched from the platform; do not re-paraphrase it.
- **Counts** — must match the rows listed in the corresponding section.
- **"Gaps — please review"** — be specific. State the file/symbol, what changed, and the question you cannot answer.
- **Generic mode** — when no platform API is available, write the same content to `doc-update-report.md` at the repo root, prepended with the file header described in `providers/generic.md`. Replace the "companion docs PR" callout with the local `git merge` instructions from the generic provider — there is no remote PR to link to.
